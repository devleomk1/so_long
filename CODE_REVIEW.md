# so_long 코드베이스 문제점 및 개선 방안

## 1) 맵 파싱이 마지막 줄 개행(`\n`)에 의존함
- `count_max_rows_cols()`는 줄바꿈 문자를 만날 때만 `rows`를 증가시켜서, **파일 마지막 줄에 개행이 없으면 마지막 행이 누락**될 수 있습니다.
- `map_load()`도 `while (get_next_line(...) > 0)` 구조라서, `get_next_line`이 `0`을 반환하는 마지막 라인의 처리 누락 위험이 있습니다.

### 개선 제안
- `count_max_rows_cols()`에서 EOF 시점의 `tmp_cols > 0`인 경우 `rows++` 처리.
- `map_load()`는 `ret = get_next_line(...)`의 반환값이 `0`일 때도 마지막 `line`을 처리하도록 구조 변경.
- 개행 유무가 다른 맵 파일에 대한 파서 단위 테스트 추가.

## 2) 컴포넌트 개수 검증 버그 (치명)
- `check_compo_cnt()`에서 수집 아이템 개수 검증이 `maps.cnt.c`가 아니라 `maps.cnt.p`를 재검사하고 있습니다.
- 결과적으로 `COMPO_C_MAX` 제한이 실제로는 적용되지 않습니다.

### 개선 제안
- 두 번째 조건을 `game->maps.cnt.c` 기준으로 수정.
- 경계값 테스트(0개, 1개, 다수) 추가로 회귀 방지.

## 3) 맵 유효성에 경로 검증(도달 가능성) 없음
- 현재 코드는 확장자/직사각형/벽/문자 구성/개수만 검증합니다.
- 플레이어 시작점에서 모든 `C`와 `E`에 도달 가능한지 확인하는 로직(DFS/BFS flood fill)이 없습니다.

### 개선 제안
- 맵 로드 직후 복사본 맵으로 BFS/DFS 실행:
  - `P`에서 출발해 방문 가능한 타일 탐색
  - 모든 `C` 방문 가능, `E` 도달 가능 여부 확인
- 실패 시 명확한 에러 메시지 제공.

## 4) 빌드 이식성 문제 (플랫폼 고정)
- Makefile 링크 옵션이 macOS 프레임워크(`-framework OpenGL -framework Appkit`)에 고정되어 있습니다.
- 현재 환경에서도 `OpenGL/gl3.h`를 찾지 못해 빌드 실패가 재현됩니다.

### 개선 제안
- OS 감지(`uname`) 후 macOS/리눅스 링크 플래그 분기.
- Linux용 MiniLibX(예: X11, Xext, m, z) 타깃 분리.
- CI에서 최소 2개 플랫폼(또는 컨테이너 매트릭스) 빌드 확인.

## 5) 유지보수성: 일반/보너스 코드의 중복
- `src/`와 `src_bonus/`가 거의 동일한 파일 구조를 가지며, 함수명이 `_bonus` 접미사만 다른 형태로 중복되어 있습니다.
- 기능 수정 시 두 트리를 동시에 반영해야 해 결함 유입 가능성이 큽니다.

### 개선 제안
- 공통 로직을 `src/common/`으로 통합하고, 차이는 설정값/전처리 플래그/전략 함수로 분리.
- 맵 파싱, 이동/충돌, 이벤트 판정 등 핵심 로직은 단일 구현으로 유지.

## 우선순위 제안
1. **즉시 수정**: 컴포넌트 검증 버그(2), 마지막 줄 파싱 이슈(1)
2. **안정성 강화**: 도달 가능성 검증(3)
3. **개발 생산성**: 빌드 이식성(4), 코드 중복 제거(5)
